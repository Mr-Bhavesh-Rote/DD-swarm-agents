# Compliance / Adverse Due-Diligence Refactor — 2026-07-01

## Problem Statement
Feedback from the stakeholder review showed that company due-diligence reports were being written from a **financial investment** angle instead of a **US compliance / adverse screening** angle. Issues included:

- Report real estate dominated by financials, operations, and investment recommendations.
- Risk sections covered the subject's own risk-management posture rather than actual derogatory issues affecting the subject.
- Specific adverse material (e.g., ICL Group's white-phosphorus / weapons involvement) was missed.
- The research swarm and prompts were not explicitly hunting for sanctions, litigation, corruption, human-rights abuses, controversial/dual-use products, environmental harm, or adverse media.

## Goal
Reframe the system to produce **US-compliance adverse due-diligence reports**:
- Lead with the most material derogatory findings.
- Keep subject/ownership overview brief (light context only).
- Remove all investment recommendations.
- Dedicate research capacity to finding adverse issues.

### 6. Synthesizer input cap to prevent worker OOM crashes (`backend/workflow/nodes/synthesizer.py`)

**Problem:** Multiple adverse-research agents produced large narrative outputs. Passing all of that raw text to the synthesizer LLM caused the worker to be killed by the kernel OOM killer (`Killed horse pid`, `Work-horse terminated unexpectedly`).

**Fix:** Truncated each raw per-agent narrative to the most relevant portion (first 6,000 chars + last 2,000 chars when the narrative exceeds 8,000 chars) and capped the consolidated findings list to the 60 most-material claims. The synthesizer still sees the full source list and the ranked claims, so no material information is lost, but the context window no longer exceeds the worker's memory limits.

### 7. Default `planning_mode` robustness

**Problem:** The frontend was occasionally creating runs with `planning_mode: null` (stale Redux state). The backend stored the null, so the runtime planner skipped the cheap YAML template and fell through to slow, expensive AI swarm generation. Runs appeared "stuck in planning" even though the template was available.

**Fix:**
- `backend/app/schemas/contracts.py`: Added a `field_validator` that coerces `null`/`""` to `"template"` on the `RunRequest` contract so new runs can never be created with a missing planning mode.
- `backend/workflow/nodes/planner.py`: Changed the planner to treat `planning_mode: None` as `"template"` for existing runs.

## Additional Fixes (2026-07-01 afternoon) — Pipeline Quality & Source Enforcement

### 9. Stop feeding raw agent narratives to the synthesizer (`backend/workflow/nodes/synthesizer.py`)

**Problem:** The synthesizer was receiving raw per-agent narratives (which could be very long) and was asked to "write comprehensively." This both caused OOM crashes and invited the writer to hallucinate unsupported claims (30% faithfulness on the ICL Group report).

**Fix:** Removed the raw-narrative block from the synthesizer's shared context. The writer now receives only the structured, consolidated findings list plus the global source list. This eliminates the evidence-blind truncation regression and forces the writer to ground every claim in a specific finding.

### 10. Severity-ranked findings cap (`backend/workflow/nodes/synthesizer.py`)

**Problem:** The 60-claim cap was applied in arbitrary order, so a high-severity finding (e.g., a white-phosphorus contract) could be dropped while lower-value claims survived.

**Fix:** Added `_finding_priority` that scores by confidence and boosts priority derogatory categories (sanctions, weapons/dual-use, corruption, etc.). Findings are sorted by this priority before the 60-claim cap.

### 11. Stricter synthesizer grounding prompt (`backend/app/core/prompts.py`)

**Fix:** Rewrote `SYNTHESIZER_SYSTEM` to instruct the writer that it may only write claims directly supported by the findings, must cite global source ids, and must not invent facts, dates, or figures.

### 12. Faithfulness threshold / blocking quality gate (`backend/app/core/config.py`, `backend/workflow/runner.py`, `backend/app/db/models.py`)

**Fix:** Raised `revision_min_faithfulness` from 0.70 to 0.85. After the graph finishes, if the final report's faithfulness score is still below 0.85, the run is marked `needs_review` instead of `done`, preventing a low-grounding report from being auto-published.

### 13. Dedicated compliance source tools (`backend/workflow/tools.py`, `config/agents.company.yaml`)

**Problem:** Required sources (OFAC, EPA ECHO, OCCRP, FPDS, etc.) were prompt-instructed but not code-enforced; agents could satisfy the instruction with a generic `web_search`.

**Fix:** Added explicit tool names in `workflow/tools.py` for each mandated source: `ofac_sdn_search`, `ofac_nonsdn_search`, `bis_entity_list_search`, `un_sanctions_search`, `eu_sanctions_search`, `fpds_search`, `usaspending_search`, `occrp_search`, `epa_echo_search`, `osha_search`, `who_profits_search`, `violation_tracker_search`, `pacer_search`. Each tool performs a targeted site search and records provenance. Updated `config/agents.company.yaml` so the adverse-research agents are assigned these tools.

### 14. Domain-tagged source completion gate (`backend/app/schemas/contracts.py`, `backend/workflow/config_loader.py`, `backend/workflow/nodes/research.py`)

**Problem:** A name-keyed gate would silently fail in AI-tailored mode because the orchestrator invents agent names per run.

**Fix:** Added an `AgentDomain` enum (`overview_ownership`, `sanctions_legal`, `adverse_conduct`, `adverse_media_esg`, `pep_ownership_risk`) to `AgentSpec`. Re-keyed `REQUIRED_TOOLS_BY_DOMAIN` and the per-agent completion gate by domain. Added `domain` to `config/agents.company.yaml` and propagated it through `config_loader.py` for both template and imported plans. The research agent now gets up to 2 extra turns if any required source for its domain has not been attempted.

### 15. Swarm domain coverage for AI-tailored plans (`backend/app/core/prompts.py`, `backend/workflow/nodes/planner.py`, `backend/app/api/runs.py`)

**Fix:** Updated the orchestrator prompt to require every agent to emit a `domain` tag and to cover all five domains. Added `_ensure_domain_coverage` in `planner.py` (and mirrored in `runs.py` for plan-preview generation) that injects a default agent for any missing domain before the swarm is dispatched.

### 16. Source manifest in the final report (`backend/workflow/nodes/renderer.py`)

**Fix:** Added a `source_manifest` object to the final report that records, for every required source tool, whether it was attempted and which domain required it. The manifest is rendered as a "Sources Queried" appendix in both the markdown and PDF exports.

### 17. Structured, length-capped Subject Overview (`backend/workflow/nodes/synthesizer.py`)

**Problem:** The "Subject Overview" section prompt instruction (max 2 short paragraphs) was frequently violated, producing shareholder tables and multi-paragraph UBO chains.

**Fix:** Added a `SubjectOverview` Pydantic model with fixed, `max_length`-constrained fields. The synthesizer uses this schema for the `subject_overview` section, then renders the fields to a short markdown list. This makes it structurally impossible to insert tables or long ownership chains in that section.

### 18. Frontend support for new status and source manifest (`frontend/src/types.ts`, `frontend/src/routes/Dashboard.tsx`, `frontend/src/routes/RunDetail.tsx`, `frontend/src/routes/ReportViewerPage.tsx`, `frontend/src/components/ReportViewer/ReportViewer.tsx`)

**Fix:** Added `AgentDomain` and `SourceManifestEntry` types, colored `needs_review` as warning in the dashboard and run detail, enabled the View report / Export buttons for `needs_review`, and displayed the Sources Queried manifest in the Final report tab.

## Operational Debugging Fixes (2026-07-01 evening) — Stuck Runs & Missing Agent

### 19. Synthesizer Pydantic schema serialization error (`backend/workflow/nodes/synthesizer.py`)

**Symptom:** Worker crashed with `TypeError: BaseModel.model_dump_json() missing 1 required positional argument: 'self'` followed by `TypeError: can only concatenate str (not "dict") to str`.

**Root cause:** `_draft_subject_overview` was calling `SubjectOverview.model_dump_json()` as a class method and then trying to concatenate the resulting dict to a string.

**Fix:** Added `import json` and changed the prompt assembly to use `json.dumps(SubjectOverview.model_json_schema(), indent=2)` so the schema is serialized to a string before concatenation.

### 20. Duplicate prepared statement error in the LangGraph checkpointer (`backend/workflow/runner.py`)

**Symptom:** Runs hung in `planning` / `researching` / `verifying` with no worker progress after the job was picked up. Worker logs eventually showed `sqlalchemy.exc.ProgrammingError: (psycopg.errors.DuplicatePreparedStatement) prepared statement "_pg3_6" already exists`.

**Root cause:** The LangGraph `PostgresSaver` checkpointer opens its own direct `psycopg` connection. Across worker restarts and pooled connections, psycopg 3 reused prepared statement names, causing the statement to collide and the run to hang silently.

**Fix:** Set `prepare_threshold=None` (not `0`) on the `psycopg.connect(...)` call in `_checkpointer()` and on the SQLAlchemy `sync_engine`/`async_engine` in `app/db/session.py`. In psycopg 3, `0` means "prepare every statement immediately", which is the opposite of the intent; `None` disables prepared statements entirely and prevents the duplicate-name collisions.

### 21. `.env` inline comment parsing failure (`backend/.env`)

**Symptom:** Worker container failed to start after the previous fix, reporting Pydantic validation errors such as `Input should be a valid integer [type=int_parsing, input_value='6000 # source text shown to the verifier per claim']`.

**Root cause:** The `.env` file contained values with trailing inline comments (e.g., `VERIFIER_SOURCE_CHARS=6000 # ...`). When the Docker image was built with `COPY . .`, these comment strings were baked in as environment variables, and Pydantic tried to parse them as numbers.

**Fix:** Stripped inline comments from the `.env` file so only the key/value pairs remain. (Note: `docker restart` does not re-read `--env-file`; containers must be recreated to pick up changes to the host `.env` file.)

### 22. Missing `pep_ownership_risk_researcher` task in the standard template (`config/tasks.company.yaml`)

**Symptom:** New company runs showed only 4 research agents instead of 5, and the UI displayed `Research swarm — 1/4 completed`. The `pep_ownership_risk` domain was not covered.

**Root cause:** The `pep_ownership_risk_researcher` agent was added to `config/agents.company.yaml`, but the loader in `workflow/config_loader.py` only includes agents that are referenced by a task in `config/tasks.company.yaml`. No task referenced the new agent, so it was silently dropped from the standard plan.

**Fix:** Added a new `pep_ownership_risk_research` task in `config/tasks.company.yaml` that binds to `pep_ownership_risk_researcher`. Verified the loader now returns all 6 agents:

```
['overview_ownership_researcher', 'sanctions_legal_researcher',
 'adverse_conduct_researcher', 'adverse_media_esg_researcher',
 'pep_ownership_risk_researcher', 'risk_aggregator']
```

### 23. Worker container stuck / non-responsive UI cancel button

**Symptom:** Runs were stuck in `planning` or `researching` and the UI cancel button had no effect. Manual cancellation required database intervention.

**Root cause:** When the worker is hung on a blocking call or database error, the RQ cancellation signal cannot be processed because the worker process is not responsive. The Redis job remains in the `StartedJobRegistry` and the UI cancel times out.

**Fix:** When manual cancellation is required, the recovery procedure is:

1. Update the run status in Postgres: `UPDATE runs SET status = 'cancelled' WHERE id = 'RUN_ID';`
2. Delete the stuck job from Redis: `redis-cli DEL rq:job:<job_id>` (or flush all RQ job keys if the job ID is unknown).
3. Restart the worker container so it re-joins the queue in a clean state.

## Follow-up corrections (2026-07-02)

### 25. Correct `prepare_threshold` value (`backend/workflow/runner.py`, `backend/app/db/session.py`)

**Symptom:** Even after the initial change, the worker still reported `DuplicatePreparedStatement` errors (`_pg3_7` etc.) and runs continued to hang.

**Root cause:** The initial change set `prepare_threshold=0`, which in psycopg 3 means "prepare every statement immediately" rather than "disable prepared statements". This made the problem worse.

**Fix:** Changed both `_checkpointer()` in `backend/workflow/runner.py` and the SQLAlchemy engines in `backend/app/db/session.py` to use `prepare_threshold=None`. The Docker image was rebuilt with `--no-cache` to guarantee the change was applied, and the backend/worker containers were recreated.

### 26. Langfuse prompt lookup retries when credentials are invalid (`backend/app/core/prompts.py`)

**Symptom:** Worker logs were continuously flooded with `Error while fetching prompt 'research_agent_system-label:production': status_code: 401 ...` and `Failed to export span batch code: 401, reason: Unauthorized`. These retries slowed the worker and obscured real progress.

**Root cause:** `get_template()` called `langfuse.get_client()` on every prompt fetch even when the Langfuse public/secret keys were empty or invalid, causing repeated failing HTTP requests.

**Fix:** Added an early return in `backend/app/core/prompts.py` that uses the local template immediately when `settings.langfuse_enabled` is false. Valid Langfuse credentials are still honored when configured; with empty/invalid credentials the worker now falls back to local templates without network calls.

The reports for `cd80e857-aaee-439b-86cf-3d1f889534c5` were regenerated from the checkpoint state and persisted manually while the new image was being built.
